## 2. 型安全規約

**原則**: 型安全規約は本セクションで方針を述べ、**具体的な検出と強制は CI が行う**。

- **`any` 型を絶対に書かない** (最優先5原則 §2)
- **`unknown` 型も書かない** (最優先5原則 §2、tests/scripts のみ例外)
- **型チェック抑止コメント (`@ts-ignore` / `# type: ignore` 等) を禁止する**
- **抑止コメントは description (10 文字以上) 付きで限定的に許可** (テスト等の意図的な型エラー検証用途のみ)
- **外部入力 (API レスポンス、AI 出力、ユーザー入力) は必ずスキーマでランタイム検証する** (TypeScript なら Zod、Python なら pydantic 等)
- **手動 if バリデーションは使わない** (スキーマで一元管理)

CI で `error` レベルで強制されるルールの詳細は §「CI で強制されているルール (参考)」を参照。

### 2.1 unknown 型禁止の理由

`unknown` を許可すると「いったん unknown で受けて narrow し忘れる」事故が起きやすい。型不明な外部入力は最初からスキーマで具体型に narrow するのが本筋。例外は `**/*.test.ts`, `**/*.spec.ts`, `tests/**/*.ts`, `scripts/**/*.ts` のみ。

## 5. ドキュメント運用ポリシー

### 5.0 ローリング運用 — フェーズ完了で「正本に統合 → サマリー差し替え」

ドキュメントは **正本 + 現在進行中のフェーズの指示書 + サマリー 1 件** の 3 種類で運用する。完了したら正本に統合してアーカイブ / 削除し、**過去のフェーズ作業ノートを溜め込まない**。

#### 構成 (常にこの 3 つだけ)

1. **設計正本** (1 件、不変): 例 `docs/architecture/cats-breeding-simulator-architecture.md` (or HTML)
2. **現在進行中フェーズの指示書** (0〜1 件): 例 `docs/design/phase_N_specification.md`
3. **直近完了フェーズのサマリー** (1 件、ローリング): 例 `docs/design/CURRENT_PHASE_SUMMARY.md` — 次フェーズエージェントが起動時に読む唯一の引き継ぎ

#### フェーズ完了時のフロー (必須)

1. 実装完了
2. **正本に該当フェーズの変更を記述** (これがないと完了とみなさない)
3. 完了フェーズの **新サマリー** を作成 (1 件)
4. 前回サマリーを **削除**
5. 指示書 (`phase_N_specification.md`) は **削除またはアーカイブ移動**
6. KICKOFF / NOTES / AUDIT 等の作業ノートも **全部削除**

#### 禁止パターン

- フェーズ完了後に指示書 / 作業ノートを溜め込む
- 「KICKOFF」「SUMMARY」「NOTES」「AUDIT」系の **複数件同居** (= サマリーは常に 1 件まで)
- 進捗・調査・分析の md 化 (チャット返信 / PR description / Task で済ます)

**Why**: ドキュメント分散 → 単一ソース・オブ・トゥルース喪失 → 後から読んだ AI / 人間が認識ズレ → 設計違反コード混入の温床。

### 5.1 設計書の更新義務

- 各フェーズ完了時、設計正本に **必ず** 内容を統合
- 統合せずにフェーズ完了とみなすのは禁止 (= サマリー差し替えだけでは不十分)
- 実装 PR と設計書更新は **同 PR でも別 PR でも可** (柔軟に判断、強制しない)

### 5.2 変更種類別の更新対象

| 変更種類 | 更新対象 |
|----------|----------|
| アーキテクチャ変更 | 設計正本 |
| API 変更 | `docs/API.md` (例) |
| 領域固有変更 | 該当領域の設計書 |

### 5.3 新規ファイル作成は最終手段

新規ファイルは「作業の副産物」ではなく「設計上の追加物」。実装・修正・検証でファイルを増やす前に、以下を必ず確認する。

1. 既存ファイルへの追記・修正で済まないか
2. 既存テスト / スクリプト / helper / util / service にケース・オプションを追加して統合できないか
3. そのファイルが恒久的に必要か、一時的な調査用か

新規作成が必要な場合は、PR 説明に **責務 / 既存統合しなかった理由 / 恒久 or 一時の別 / 実行 or 参照経路 / 削除条件** を必ず記載する。

一時調査ファイルを `src/` 配下やリポジトリ直下に置かない (`.tmp/` / `tmp/` / `scratch/` を使い、`.gitignore` する)。テストはバグ修正ごとに新ファイルを作らず、既存テストファイルにケース追加するのが原則。

### 5.4 scripts/ ディレクトリの運用

`scripts/` 配下に置くスクリプトは **`scripts/README.md` の登録表に必ず追記**する。追記しないスクリプトは追加してはならない。用途分類は `dev/` `check/` `migrate/` `ci/` `maintenance/` `one-shot/`、one-shot は冒頭に削除予定日を記載する。詳細は [`scripts/README.md`](./scripts/README.md) を参照。

### 1. 計算モードの分離 (シミュレーター正本 §2)

- **normal (通常モード)**: 通常UI/API。親の表現型・猫種・性別から **確定できる遺伝子条件のみ** 使用。
- **explicit_carrier (明示キャリアモード)**: ユーザーが親の隠れキャリアを明示した場合のみ、未表現キャリア由来カラーを通常結果に含めてよい。
- **carrier_exploration (全キャリア探索モード)**: 研究/検証用。結果は `carrier_exploration_results` として通常結果と **完全分離** する。通常結果に混ぜてはならない。

### 2. 優性表現型のヘテロ未確定ルール (シミュレーター正本 §2.4) ★最重要

**表現型に現れている優性形質は 優性/- として扱い、ホモ接合に固定しない。ただし、表現型から直接要求されない潜在キャリアは、通常モードでは勝手に展開しない。**

- **カテゴリA (優性ヘテロ未確定 / 通常モードでも展開する)**: タビー `A/-`、濃色 `D/-`、シルバー `I/-`、マッカレル `Mc/-`、ティックド `Ta/-`。優性形質が見えていてもホモ/ヘテロは不明なため `{X/X, X/x}` の両方を計算対象にする。
  - 例: `Silver Tabby` の父は `I/-`(→非シルバーの子 Brown 系)・`D/-`(→希釈の子 Blue 系)・`A/-` を生み得る。
- **カテゴリB (劣性固定)**: `d/d`(Blue/Cream), `a/a`(Solid), `cs/cs`(Point), `cb/cb`(Sepia), `cb/cs`(Mink), `i/i`(非シルバー), `s/s`(白斑なし) は固定。
- **カテゴリC (潜在キャリア / 通常モードでは展開しない)**: `B/b`(チョコ), `B/bl`(シナモン), `C/cs`(ポイント), `C/cb`(セピア)。明示キャリアまたは血統・産子履歴がある場合のみ explicit_carrier で扱う。
- **S (白斑) は特例**: 不完全優性で `S/S`(Van) と `S/s`(-White) は表現型で区別でき、入力白斑レベルから接合性が確定する。よって X/- 展開せず CSV 値のまま固定する (逆引きMAPの Van 誤マッチも回避)。
- 実装: [`master_data.py`](./cat_breeding_simulator/master_data.py) `_build_normal_parent_genotypes` がカテゴリA座をヘテロ展開し、CSV逆引きに無い遺伝子型は [`engine.py`](./cat_breeding_simulator/engine.py) `_construct_fallback_name` が標準表現型から構築命名する (未分類ゼロを保証)。

### 3. 不可逆ルール (劣性ホモは優性へ戻らない / シミュレーター正本 §4)

子の遺伝子型はメンデル分離で決まり、以下は **絶対に生成されない**:

- `d/d × d/d` → 濃色 `D/-` (例: Cream Tabby-White × Dilute Calico から濃色 `Calico` は出ない。`Dilute Calico` のみ)
- 非オレンジ `a/a × a/a` → タビー `A/-`
- `cs/cs × cs/cs` / `cb/cb × cb/cb` → フルカラー `C/-`
- `w/w × w/w` → `White` / `s/s × s/s` → 白斑 / `i/i × i/i` → Silver・Smoke / `ta/ta × ta/ta` → Ticked / `mc/mc × mc/mc` → Mackerel / `wb/wb × wb/wb` → Chinchilla/Shaded/Golden系
- 通常XYオス (`o/Y` または `O/Y`) に Tortie / Tortoiseshell / Calico / Blue Cream / Lilac Cream を出してはならない。

### 4. 出力・診断ルール (シミュレーター正本 §6〜§8)

- CSVに存在する名前でも、親の表現型から導出不能・性別矛盾・猫種文脈なし・白斑レベル未明示なら通常結果に出さない。
- 分類不能な子の遺伝子型を **黙って捨てて再正規化しない**。`unmatched_probability` 等で保持し検出可能にする (実装: `CoatColorCalculator.calculate_report`)。
- 丸め後 `0.0%` になる項目は通常出力に出さない。
- 表示合計は100%。ただし未分類を捨てた再正規化100%は不合格。

### やってはいけないこと一覧 (運用正本 §8.2)

- 仕様正本と異なるキャリア自動展開
- `None` (分類不能) をスキップして残りを100%に再正規化する実装
- CSVにある名前なら何でも出す実装
- `0.0%` 結果の通常表示
- テストを期待値側で雑に変更して通すこと (仕様変更に伴う期待値更新は理由を明記する)
- 計算ロジック修正だけのタスクで `Dockerfile` / `.github/workflows/` / Cloud Run・GCP 設定を変更すること (運用正本 §3.3)

## AI 推論品質ガイドライン

AI エージェントが矛盾や誤推論を防ぐための品質基準。複雑な推論・判断が必要なタスクにおいて**必ず適用**する。

### 1. Self-Verification (出力前矛盾チェック)

**必須対象**: 複数の選択肢・解釈を提示する場合

**実行手順**:
1. 出力内容全体を確認し、以下の2点を矛盾チェック
   - **一貫性**: 書いた内容全体が矛盾なく一貫しているか
   - **指示適合性**: ユーザーの指示と矛盾していないか
2. 矛盾が見つかれば訂正してから出力
3. **1つの回答内で複数の矛盾する解釈を並べない**

### 2. Uncertainty 表示 (推論事実の明示)

**基準**: 推測・推論が発生した時点で明示

**推論が起こるケース**:
- ドキュメントに明記されていない仕様の判断
- ユーザー指示が曖昧で解釈が必要
- コード内容から仕様を逆算する推測

**推論時の表現例**:
- ✅ 「これはコード内容からの推測ですが、〜」
- ✅ 「推論: ユーザーの意図は〜」

**避けるべき表現**:
- ❌ 「明らかに〜」 (推測を事実として提示)
- ❌ 「当然〜」 (確信がない判断)

### 3. Backtracking (自己訂正)

**現実的制限**: ガードレール効果は限定的。「誤りに気づいたら停止する」と記載しても、実際には実装を止めず最後まで進行する可能性が高い。

**代替案**:
- 実装完了後、出力前に矛盾が無いか最終チェック (Self-Verification で補完)
- 複雑なタスクではユーザーに選択肢を提示し、判断を委譲する

## 作業完了時セルフチェック

フェーズ・チケットの完了報告を行う前に、以下を確認する:

- [ ] 指定されたフェーズの完了条件を全て満たしているか
- [ ] 既存テストが全て通るか
- [ ] 新規コードにユニットテストを追加したか
- [ ] データ構造の破壊的変更を行っていないか
- [ ] 設計書の禁止事項に抵触していないか
- [ ] ログ・ドキュメントを `<PRIMARY_LANGUAGE>` で記述したか

## 困った時

- 設計の意図が分からない → 設計正本を読む
- フェーズの範囲が不明確 → 該当フェーズの指示書を読む
- 既存コードの構造が分からない → 勝手に推測せず、ユーザーに確認する
- 新しい設計判断が必要 → 勝手に決めず、ユーザーに確認する

**「勝手に決めない」**。これが最重要 (最優先5原則 §1)。
