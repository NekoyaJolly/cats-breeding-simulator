### 2.1 マスタCSVの読み込み契約 (Fail-Fast)

マスタCSVはモジュール import 時 (= サーバ起動時) に読み込む。**欠落・破損時は空データで起動を続けず、`RuntimeError` で即座に停止する (Fail-Fast)**。これにより Docker へのコピー漏れ・文字化け・配置ミスを「本番での無言の機能不全 (全リクエストが謎のエラー)」ではなく「起動失敗 + 原因ログ」として即座に検知する。

| ローダー (モジュール) | 対象CSV | 欠落/破損時 |
|---|---|---|
| `master_data._load_color_base_loci` / `_load_color_definitions` | `cat_color_genetic_map.csv` | `RuntimeError` |
| `master_data._load_breed_filters` | `cat_breed_genetic_map.csv` | `RuntimeError` (旧: ハードコードfallback) |
| `breed_color_policy._load_policy_rows` | `cat_breed_color_policy.csv` | `RuntimeError` |
| `color_master._load_master_rows` | `cat_color_master.csv` | `RuntimeError` |
| `display_alias_map._load_map_rows` | `cat_color_display_alias_map.csv` | `RuntimeError` |

- 検出する「破損」: ①ファイル欠落 ②文字コード不正 (`UnicodeDecodeError`) / CSV構造不正 (`csv.Error`) ③**`DictReader` が例外を出さない破損** — 空ファイル・ヘッダ不正・必須列欠落 (`CoatColor`/`ColorId`/`AliasId`/`Breed`/`PolicyStatus` など) により**有効行が0件**になるケース。③は読み込み後の件数チェックで検知する (`_load_breed_filters` は Munchkin 補完より前で判定)。
- 検証: [`tests/test_csv_fail_fast.py`](../../tests/test_csv_fail_fast.py) (欠落・文字コード破損・ヘッダ不正・空ファイル・Dockerfile同梱漏れを網羅)。
- 効果: Cloud Run デプロイ後のヘルスチェックでコピー漏れが失敗として検知される (旧挙動は起動成功 → 全リクエスト異常で気付けない)。
- 前提: テスト/ローカル実行時も対象CSVが配置されていること (`docs/architecture/` 配下、または cwd)。

## 4. API運用仕様

### 4.1 エンドポイント

```text
POST /api/v1/calculate
POST /api/v1/reverse-lookup
```

### 4.2 通常モードリクエスト

```json
{
  "sire_color": "Silver Tabby",
  "dam_color": "Blue Pt Tabby-White",
  "breed": "Any",
  "mode": "normal"
}
```

### 4.3 明示キャリアモードリクエスト

```json
{
  "sire_color": "Silver Tabby",
  "dam_color": "Blue Pt Tabby-White",
  "breed": "Any",
  "mode": "explicit_carrier",
  "sire_carriers": {
    "C": "C/cs"
  },
  "dam_carriers": {
    "C": "C/cs"
  }
}
```

### 4.4 レスポンス標準

実装 (Phase 1): 従来の `status` / `parameters` / `results` に加え、`mode` と `diagnostics`
(opened_loci / closed_loci / assumptions / 確率診断) を追加 (additive, 後方互換)。

```json
{
  "status": "success",
  "mode": "normal",
  "parameters": { "sire_color": "...", "dam_color": "...", "breed": "Any", "mode": "normal" },
  "results": [],
  "diagnostics": {
    "opened_loci": ["A", "D", "I", "Mc", "Ta"],
    "closed_loci": ["B", "C", "Wb"],
    "assumptions": ["A/D/I/Mc/Ta: 優性ヘテロ未確定として X/- 展開", "..."],
    "matched_probability": 100.0,
    "unmatched_probability": 0.0,
    "unmatched_genotype_count": 0
  }
}
```

> carrier_exploration (Phase 2, 実装済): `results` (normal 確定結果) に加え、`carrier_exploration_results`
> (片親の劣性発現に対し相手がキャリアの場合の条件付きシナリオ配列) を返す。事前確率は掛けない。
> 各シナリオ: scenario / label / assumed_carriers / probability_basis / prior_probability_applied / results / new_colors。

### 4.5 目標カラーから探す (逆引きMVP)

`POST /api/v1/reverse-lookup` は、登録猫または自舎カラー構成をリクエストで受け取り、目標カラーが
成立し得る交配候補を返す。保存層はAPI側に持たず、MVPのフロントエンドではブラウザのローカル保存を
抽象リポジトリ経由で使う。将来DB化する場合は保存層を差し替え、探索ロジックは同API契約を維持する。

`target_color` は必須、`target_sex` (`male` / `female`) は任意。性別指定がある場合は、該当性別の
子猫結果だけを目標確率として集計する。
登録猫の毛色入力は通常シミュレーターと同じ名前解決を使い、父母の性別と `SexRestriction` が矛盾する
場合は 422 を返す。保存済みデータやAPI直送の不整合を候補なしとして扱ってはならない。

レスポンスは、確定確率、条件付き最大確率、確認が必要な条件、推奨検査、座位別根拠、目標カラー以外に
生まれる可能性のあるカラーを含む。候補が見つからない場合も、目標カラーに必要な条件と現在確認できない
条件を返す。

### 4.6 リター実績から推定 (MVP)

`POST /api/v1/litter-inference` は、父猫・母猫の表現型と観察された子猫群のカラー・性別を受け取り、
観察実績を説明できる親遺伝子型候補を返す。保存層はAPI側に持たず、MVPのフロントエンドではフォーム入力のみを扱う。

入力カラーは通常シミュレーターと同じ名前解決を使い、親候補生成・Punnett 分離・子猫遺伝子型構築は
`CoatColorCalculator` の共通処理を使う。色名・遺伝子座の読み込み経路を新設してはならない。

レスポンスは、確定、条件付き確定、推定、未確認、矛盾、警告、推奨検査を分けて返す。Red / Cream 系の
A座位判定、Calico / Tortie 系の白斑判定は断定せず警告として扱う。

### 4.7 互換性方針

既存APIレスポンスを壊す変更は原則禁止する。

必要な場合は以下のどちらかとする。

- 新フィールドを追加する
- `/api/v2/calculate` を追加する

## 5. テスト方針

### 5.1 必須テスト

以下は必須とする。

| テスト | 目的 |
|---|---|
| test_irreversible_rules.py | 不可逆ルール検証 |
| test_cross_130_204_output_validity.py | 130×204の回帰テスト |
| test_csv_fail_fast.py | CSV Fail-Fast・マスター契約検証 |
| test_calculator_api.py | API互換性検証 |

### 5.2 不可逆ルールテスト

最低限、以下を検証する。

- d/d × d/d から濃色が出ない
- a/a × a/a から非オレンジタビーが出ない
- o/Y × o/o からRed, Cream, Tortie, Calicoが出ない
- cs/cs × cs/cs からフルカラーが出ない
- w/w × w/w からWhiteが出ない
- s/s × s/s から-Whiteが出ない
- ta/ta × ta/ta からTickedが出ない
- i/i × i/i からSilver/Smokeが出ない
- 通常XYオスにTortie/Calico/Blue Creamが出ない

### 5.3 130×204回帰テスト

入力:

```json
{
  "sire_color": "Silver Tabby",
  "dam_color": "Blue Pt Tabby-White",
  "breed": "Any",
  "mode": "normal"
}
```

禁止語:

- Mink
- Sepia
- Point
- Mitted
- Bi-Color
- Bi-C
- Van
- Champagne
- Platinum
- Sable
- Chocolate
- Cinnamon
- Lilac
- Fawn
- Choco
- Lynx Point

> **仕様変更 (A: カテゴリA' → カテゴリA)**: 禁止語はいずれも B/C 座位由来 (チョコ/シナモン/ライラック/ポイント/セピア/ミンク等) の潜在キャリア＝カテゴリC のみとする。
> **Smoke / Tortie / Tortoiseshell / Calico / Solid 単独色は禁止語から除外した** (a/a 前提だが、A をカテゴリA として X/- 展開するため、両親がともに A/a のとき a/a の子が低確率で出る＝遺伝的に正しい出力)。
> Blue Cream / Lilac Cream は a/a 前提の希釈ソリッドトーティのため **メス限定** とし、オスには出さない (下記条件の「オスにTortie/Calico系がない」に含む)。

条件:

- probability_pct > 0
- matched_probability == 100.0
- unmatched_probability == 0.0
- オスにTortie/Calico/Blue Cream/Lilac Cream系がない
- a/a 前提のソリッド系 (Solid/Smoke/Tortie 等) がメスに低確率で出る (A をカテゴリA として展開する仕様変更に伴う。旧「全出力がタビー系」前提は撤回)
- 合計確率が100%

## 8. Codex / AIエージェント運用ルール

### 8.1 指示に必ず含めること

Codexへ依頼する場合は、以下を必ず書く。

- 修正対象ファイル
- 触ってはいけないファイル
- 実行するテスト
- 合格条件
- 130×204回帰テストを通すこと
- ソースコード内コメントは日本語

### 8.2 禁止

AIエージェントに以下を許可しない。

- 計算ロジック修正中のCloud Run設定変更
- 理由のない依存関係追加
- 仕様書と異なるキャリア自動展開
- `None`をスキップして再正規化する実装
- CSVにある名前なら何でも出す実装
- 0.0%結果の表示
- テストを期待値側で雑に変更して通すこと

### 8.3 報告フォーマット

AIエージェントは作業後に以下を報告する。

1. 根本原因
2. 修正ファイル
3. 修正関数
4. 仕様正本との対応
5. 追加テスト
6. 実行コマンド
7. テスト結果
8. Cloud Run / CI/CDへ影響があるか
